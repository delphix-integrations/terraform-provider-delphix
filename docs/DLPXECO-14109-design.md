# Feature Design: DLPXECO-14109

**Jira**: [DLPXECO-14109](https://perforce.atlassian.net/browse/DLPXECO-14109) — Security: Remediate 23 transitive dependency vulnerabilities (CVE batch) in terraform provider
**Status**: Proposed
**Source branch**: `vuln-fix` (in-place; no worktree)

---

## Summary

This change bumps four transitive Go modules in `go.mod` / `go.sum` to remediate 23 CVEs (9 Critical, 3 High, 9 Medium, 2 Low) flagged on `terraform-provider-delphix` v4.3.0 by the internal Security Check scan. The modules — `golang.org/x/crypto`, `golang.org/x/net`, `google.golang.org/grpc`, `golang.org/x/sys` — are pulled in via `github.com/hashicorp/terraform-plugin-sdk/v2 v2.33.0` and `github.com/hashicorp/terraform-plugin-log v0.9.0` and are not directly imported by any provider source file. The fix is dependency-only: no provider behaviour, no schema, no resource lifecycle, and no public API surface changes. The provider must continue to build cleanly on Go 1.25, `make test` must keep its current pass count, and a post-bump Security Check must report **0 of the original 23** CVEs as still applicable. The bumped versions stay marked `// indirect`; no `replace` directives are added; the direct SDK and plugin-log dependencies are intentionally left unchanged (per Non-Goal NG1).

## Architecture Changes

None. This is a dependency-version bump only. No Terraform schema field is added, removed, deprecated, or re-typed. No Engine API JSON payload struct (`models.go`) changes. No new env var, no new provider block field, no new resource. The `ResourcesMap` in `provider.go` remains byte-identical.

## Platform Behavior Notes
<!-- Non-obvious behaviours documented in CLAUDE.md / architecture.md that this change might brush against. Marked "Affects" only if the bump materially changes the behaviour; "N/A" otherwise. All N/A entries are kept so reviewers see they were considered. -->

- **Async job polling (`PollJobStatus`, 20s interval, terminal states in `commons.go`)** — N/A. This logic is in `internal/provider/utility.go` and does not import any of the bumped modules directly. The DCT API call layer underneath uses `dct-sdk-go` over `net/http` (stdlib), not `grpc`.
- **TLS configuration (`InsecureSkipVerify` switch, `security.go`)** — Affects (low risk). `golang.org/x/crypto v0.52.0` and `golang.org/x/net v0.55.0` ship cumulative TLS / HTTP/2 hardening. The provider's own `tls.Config` is built from `crypto/tls` (stdlib), not from `golang.org/x/crypto`, so the direct surface is unchanged; however, the indirect HTTP/2 client used by `dct-sdk-go` over `golang.org/x/net/http2` may negotiate differently with very old DCT endpoints. Validation phase to re-check `DCT_HOST_SCHEME=https` flows against a v29 engine.
- **Engine API session cookie + login flow (`engine_api.go`)** — N/A. Uses stdlib `net/http`. No grpc, no `golang.org/x/net` direct symbol.
- **Destructive VDB update cycle (disable → update → enable, `isDestructiveVdbUpdate` map)** — N/A. Pure DCT REST flow; orthogonal to dependency bump.
- **`tflog` structured logging with `[DELPHIX]` prefix** — N/A. `terraform-plugin-log v0.9.0` is unchanged; we are only bumping its transitive `x/net` / `x/sys` floors. `tflog` public API is untouched.
- **`SecureString` memory-zeroing for sensitive fields (`security.go`)** — N/A. Pure Go; no crypto-library dependency.
- **Tag diff suppression (`CustomizeDiffTags`, `HandleRawConfigReadContext`)** — N/A. Schema-layer logic; unaffected.
- **`ignore_tag_changes` default-true semantics** — N/A. Same as above.
- **Acceptance test gate (`TF_ACC=1`, `testAccPreCheck`)** — N/A for the merge gate; relevant only for the on-demand pre-release acceptance run noted in EC-6.
- **goreleaser GPG-signed checksums + multi-platform build** — Affects (verification only). Validate phase runs `make release` dry-run to confirm all cross-compile targets still link. No expected behavioural change; pure-Go modules link the same on every OS.

## Open Questions / Risks

<!-- Carried forward from vision Risks and refined with what was learned during this design pass. Blocking items (escalation paths) sit at the top. -->

- **R: `terraform-plugin-sdk/v2 v2.33.0` (or its transitive `terraform-plugin-go v0.22.0`) fails to compile against `grpc v1.79.3`** — Mitigation: do `go get` + `make build` as the first action in the implement phase. If the compile error is in any package under `internal/provider/`, treat it as a true semantic break; if it is in an indirect package (likely under `vendor/`-style cache or `mod/cache`), inspect the specific symbol. Per EC-4, halt and open a follow-up SDK-bump ticket rather than rolling back the security fix or papering over with `replace`. *Likelihood: Low; Impact: High.*
- **R: `go mod tidy` cascades unrelated bumps** (e.g. raises `golang.org/x/text`, `golang.org/x/tools`, `google.golang.org/genproto/googleapis/rpc`, `google.golang.org/protobuf`) — Mitigation: run tidy once, diff `go.mod` and `go.sum`, accept only the four required bumps plus their transitive floor bumps that are strictly required to compile. Any unrelated bump gets reverted via `go mod edit -require=<m@v>` + re-tidy. Validation doc captures the final dependency delta. *Likelihood: Medium; Impact: Low.*
- **R: HTTP/2 / TLS behaviour change in `golang.org/x/net v0.55.0` or `golang.org/x/crypto v0.52.0` causes a regression against a very-old DCT engine** — Mitigation: validate phase runs the existing unit tests (the merge gate) and notes the consideration; acceptance tests against staging cover the live-engine path before tagging the next release per EC-6. Release notes (doc-updates phase) will call out any cipher-suite floor or ALPN change picked up between v0.45 and v0.52 / v0.47 and v0.55. *Likelihood: Low; Impact: Medium.*
- **R: A bumped module is yanked / unavailable on `proxy.golang.org`** (EC-5) — Mitigation: fall back to the next-higher minor version that still clears the original 23 CVEs and document the substitution in the validation doc and PR description. Do not silently downgrade. *Likelihood: Very Low; Impact: Low.*
- **R: A test file references a renamed/removed symbol from a bumped module** (EC-3) — Mitigation: apply the minimum syntactic rename only; do not change assertion logic; document the edit explicitly in the PR. The most-likely candidate is anything referencing `grpc.WithInsecure` (deprecated since grpc 1.50, removed in 1.79.x line); a quick grep over `internal/provider/*_test.go` for `grpc.` will confirm whether any test file is affected. *Likelihood: Low; Impact: Low.*
- **R: Security Check tooling unavailable during the validate phase** (VPN / pipeline down) — Mitigation: record this in `docs/DLPXECO-14109-validation.md` Section 9 and re-run before tagging the next release. The validate phase's Overall Verdict may be `PASS WITH WARNINGS` rather than `PASS` in that case; do not fabricate FR-003 evidence. *Likelihood: Low; Impact: Medium.*
- **R: A newly-bumped indirect dependency surfaces an unrelated CVE finding** (EC-5 / ERR-5) — Mitigation: out of scope for this ticket; record in validation doc Section 9 and open a follow-up. The 23 originally-flagged CVEs are the only success measure for FR-003. *Likelihood: Low; Impact: Low.*
- **R: `go vet ./...` surfaces a new warning attributable to one of the bumped modules** — Mitigation: triage during validate phase. If the warning originates in our source, fix it; if it originates in a bumped module's own source (rare but possible — vet sometimes follows into module cache), document and ignore unless it indicates a real runtime bug. *Likelihood: Very Low; Impact: Low.*
- **Q: Should we pin the four modules with an explicit floor in our direct `require` block, or leave them in the `// indirect` block?** — Recommendation: leave indirect. Adding direct `require` lines for modules we do not import would (a) lie about the dependency graph, and (b) leak our bump-floor into downstream consumers that might prefer their own version. MVS handles the floor correctly via the `// indirect` entries. *Non-blocking — confirmed by NG1 + the "indirect-only" quality rule.*
